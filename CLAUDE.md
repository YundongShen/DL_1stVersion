## What This Project Is

**Goal:** ICLR paper on "Edit Entailment Learning" — detecting scope creep in LLM-generated code patches.

**Core idea:** A requirement change logically *entails* certain code edits. We learn a 4-entity joint embedding space (REQ, TEST, ORIG, HUNK) using InfoNCE contrastive learning, such that a hunk's geometric position relative to the other three entities reflects how necessary it is. This is the paper's main contribution — not classification, but a learned geometric structure that generalizes across repos.

**Dataset:** SWE-bench Full (2291 instances, 12 repos). Each instance has a bug report (requirement), gold patch, fail-to-pass tests, and source files.

**Collaborator:** Miaowen Dong (supervisor/co-author)

## Three-Tier Gradient (No Human Annotation Needed)

| Tier | Meaning | Source |
|------|---------|--------|
| Tier 1 | Gold hunk covered by fail-to-pass tests — highest necessity | SWE-bench structure |
| Tier 2 | Gold hunk NOT covered by any test — necessary but invisible to tests | SWE-bench structure |
| Tier 3 | LLM-generated hunk outside gold patch — scope creep, lowest necessity | Claude Haiku generation |

Tier 3 was generated by running `claude-haiku-4-5-20251001` on all 2291 instances unconstrained, extracting hunks that don't overlap with gold patch. **743/2291 instances (32.4%) yielded ≥1 Tier-3 hunk.**

## Architecture

**Single encoder:** UniXCoder (`microsoft/unixcoder-base`) with 4 type tokens:
```
<REQ>  requirement text </REQ>
<TEST> test function body </TEST>
<ORIG> source unit (function/class) from original codebase </ORIG>
<HUNK> diff hunk with 5-line context </HUNK>
```

**Loss:** MultiPairInfoNCE — 4 types of positive pairs pulled together, in-batch negatives pushed apart:
- (REQ, gold HUNK)
- (REQ, TEST)
- (ORIG code unit, HUNK that modifies it)
- (REQ, ORIG code unit in modified files)

**Inference score:**
```
EntailmentScore(hunk) = α·sim(hunk, req) + β·sim(hunk, test_mean) + γ·sim(hunk, orig_mean)
```
α, β, γ defined in `config.py` under `cfg.eval.score_alpha/beta/gamma`.

## Evaluation Results (Completed on Berzelius)

### Overall (evaluate.py on instances_full.jsonl + tier3_hunks.jsonl)
- **nDCG@5 = 0.917**
- **Tier-2 Recall = 0.837**

Metric definitions:
- nDCG@5: Tier1 weight=3.0, Tier2 weight=2.0, Tier3 weight=0.0. Measures ranking quality.
- Tier-2 Recall: Among Tier-2 hunks, what fraction appear in top-k positions (k = number of Tier-2 hunks per instance). This is the paper's hardest and most important metric — Tier-2 hunks are necessary but have no test coverage, so baseline methods miss them.

### Cross-Repo Generalization (the critical paper claim)

## Key Decisions & Why

**Why Tier-2 Recall is the paper's key metric:** Tier-2 hunks (gold, untested) are what ALL existing tools miss. These are "induced inconsistencies" — places in the original codebase that need updating because of the new requirement. Our ORIG entity is what captures this: a hunk that modifies an existing function will have high sim(hunk, orig). This is the paper's unique contribution vs. simple semantic similarity methods.

**Why cross-repo eval matters:** The paper claims the 4-entity geometry is semantically meaningful, not repo-memorization. The proof is that pallets (11 training instances) gets +0.07 T2-Recall improvement — same as large repos. A memorizing model wouldn't generalize to repos with almost no training exposure.

**Why val_loss plateau is NOT a problem:** val_loss = cross-repo batch discrimination. The model can't perfectly solve "which of 32 random hunks from 12 different repos belongs to requirement A" — but it doesn't need to. nDCG measures within-instance ranking (is gold hunk ranked above scope-creep hunk), which the model does well.

**Why ablations matter:** The paper needs to show that all 4 entities are necessary. Expected result: removing ORIG drops T2-Recall significantly (ORIG is what makes Tier-2 detectable). Removing TEST drops T1 performance. Removing REQ makes everything random.
